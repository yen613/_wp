
### 執行結果

執行 `process(listA, listB)` 後的結果如下：

* **`listA` 的內容：** `[1, 2, 99]`
* **`listB` 的內容：** `[3, 4]`

---

### 為什麼？核心原因詳細拆解

在 JavaScript 中，陣列屬於物件（Object），當物件被當作參數傳進函數時，傳進去的是該物件在記憶體中的「地址（引用）」。

但為什麼 `listA` 被改變了，`listB` 卻完好如初？關鍵就在於 **「修改屬性」與「重新賦值」的本質差異**。

我們可以把記憶體的狀態拆解成三個階段來看：

#### 階段一：呼叫函數前

在全域中，`listA` 指向地址 `#001`（裡面存著 `[1, 2]`）；`listB` 指向地址 `#002`（裡面存著 `[3, 4]`）。

#### 階段二：呼叫函數並傳入參數 `process(listA, listB)`

當函數執行時，內部的區域變數 `a` 和 `b` 會分別接收這兩個地址：

* 區域變數 `a` 也指向地址 `#001`
* 區域變數 `b` 也指向地址 `#002`

#### 階段三：函數內部操作

1. **執行 `a.push(99);`（修改內部屬性）**
區域變數 `a` 順着它手上的地址 `#001`，跑到記憶體中，把裡面的陣列塞入 `99`。因為全域的 `listA` 也指向 `#001`，所以**外面的 `listA` 看到了內容的改變**。
2. **執行 `b = [100];`（重新賦值）**
這一行是最大的陷阱！當我們寫 `b = ...` 時，意思是在記憶體中**新開闢了一個獨立的空間**（假設地址是 `#003`，裡面存著 `[100]`），並強迫區域變數 `b` 轉頭去指向 `#003`。
**此時，區域變數 `b` 與原本全域的 `listB`（依然指向 `#002`）正式斷絕關係！**

#### 階段四：函數執行完畢

函數結束後，區域變數 `a` 與 `b` 被記憶體回收消失。

* `listA` 依然指向 `#001`，但裡面的資料已經在階段三被改成 `[1, 2, 99]` 了。
* `listB` 依然指向 `#002`，從頭到尾都沒有人去動過 `#002` 空間裡的資料，所以維持 `[3, 4]`。

---

### 💡 觀念總結

* **修改（如 `push`、`pop`、`shift`、或 `a[0] = x`）：** 是順著地址去動到同一個記憶體倉庫的內容，外面的變數**會**受到影響。
* **重新賦值（如 `b = ...`）：** 是直接剪斷原來的風箏線，讓函數內的變數去綁定一刻新的風箏，外面的變數**絕對不會**受到影響。
